The traditional IT planning rhythm has worked for a decade. CPUs scale with workload. RAM scales with state. IOPS scales with storage demand. Network throughput scales with users. The ratios are stable. If you add another rack of servers, you know roughly how many users it can absorb.

Inference breaks four of these assumptions. Each one breaks quietly. The numbers look familiar at first glance, but if you plan against them the way you'd plan against a database or a web tier, you will either over-provision by four times, or under-provision and discover the gap in production.

This module names each broken assumption and what replaces it. The next four modules explain the mechanics. The most important point arrives at the end: the planning practice itself has to change, not just the parameters in your spreadsheet.

## Assumption 1: you can plan from the metrics you already track

The traditional planning playbook uses metrics that any infrastructure spec sheet leads with. CPU cores. Clock speed. RAM. IOPS. Network bandwidth. For inference, those numbers won't get you a good answer.

The single most important spec for streaming inference performance is HBM memory bandwidth, and most enterprise GPU spec sheets don't even feature it in the headline. Two GPUs with the same FLOPS rating can serve very different numbers of users. An H100 and an H200 have similar compute envelopes. The H200 has about 43 percent more memory bandwidth, and serves roughly that much more decode throughput on the same model.

Beyond bandwidth, the planning conversation needs three more metrics that aren't on any traditional spec sheet. KV cache footprint per token (in kilobytes per token, often around 160 KB for a 70B model at FP8). Tokens per second per replica, which is the throughput unit that converts to "how many users can this server handle." And the precision the model will run at, which changes both memory and throughput by two times or more. None of these were in your previous planning toolkit. All of them are required now.

## Assumption 2: the workload spec contains the parameters that drive capacity

A database load test asks two questions. How many users. What query rate per user. The answer falls out of those.

For inference, the answer requires five inputs, and two of them aren't on most application specs.

The familiar ones: how many users, how many requests per user per minute, what latency SLOs.

The new ones: how long is each conversation's history (context length), and how many of those conversations are simultaneously mid-response (in-flight concurrency). These two parameters multiply, and the product is what drives the GPU memory budget. A 200-user app at 8,000-token contexts is one workload. The same 200 users at 128,000-token contexts is a workload an order of magnitude heavier. Same headcount, very different infrastructure.

This catches IT teams used to scaling on user counts. The planning question is no longer "how many users." It is "how many in-flight requests at peak, and how long are they."

## Assumption 3: working memory has graceful degradation paths

Traditional infrastructure has a memory hierarchy. Cache, RAM, SSD, disk. Workloads spill from faster tiers to slower ones when they have to, with gradually worse performance. An overloaded database starts hitting disk. A starved web app starts queueing. Performance degrades, but service continues.

Inference has no spill. The working memory of every in-flight conversation must fit in HBM, the most expensive memory tier in the system, in the same physical space as the model weights themselves. There is no swap to RAM, no offload to NVMe (without significant engineering work and a real latency hit), no graceful degradation. If HBM fills up, requests queue or fail outright.

This means a 70 GB model serving 200 conversations might actually need 200 GB of HBM, not because the model grew, but because the conversations brought their own memory and stored it in the same place the model lives. The "storage budget" and the "compute budget" are the same line item.

## Assumption 4: existing facility envelopes accommodate your workload

Most colocation facilities are designed around 8 to 15 kilowatts per rack. That has been the planning baseline for a decade, and it accommodates dense CPU racks comfortably.

A single 8-GPU server, like an HGX H100 chassis, pulls 10 to 11 kilowatts by itself. One server fills the rack envelope. Stack four of them in one rack and you are at 40 kW, well past most legacy power feeds. The latest Blackwell racks pull 120 to 140 kW and require liquid cooling as a design constraint, not a tuning choice.

GPU servers also draw power continuously at near-rated TDP, not in the bursty pattern CPU workloads have trained facilities to expect. Total facility consumption is higher than peak numbers might suggest, and cooling has to be sized for the sustained load.

The fix is not a software optimization. It is a facilities conversation, often a contract amendment, sometimes a build-out. That conversation needs to happen before any hardware lands on the loading dock.

## What this actually changes about planning

The four shifts above are the visible surface of a deeper change. Capacity planning for inference is no longer something the IT team can do alone, and it is no longer a one-time exercise.

**It is no longer solo work.** Every important number depends on a choice the AI team makes. Which model. At what precision. With what context length cap. On which serving engine. With which optimizations enabled. Two AI teams running "the same" 70B model can end up needing four times different infrastructure depending on how those choices land. You cannot estimate a workload by reading its spec. You have to ask the AI team to walk you through their stack, and you have to do that regularly enough that you catch changes before they ship.

**It is no longer a one-time exercise.** The AI ecosystem moves in months, not years. A new model release, a context window upgrade, a serving engine version bump, a quantization recipe that wasn't possible a quarter ago: any one of these changes the math. A plan that was right when you ordered hardware can be wrong by the time it arrives. The right cadence for revisiting an inference capacity plan is quarterly, not annually.

Three habits follow from this, and the IT team will need all three.

First, a standing planning conversation with the AI team. Not a one-time intake form. A recurring meeting where the AI team's roadmap (which models they're piloting, which optimizations they're testing) becomes a planning input.

Second, a willingness to revisit assumptions when conditions change. The deployment you sized last quarter for a 32K context window may need to be re-sized when the AI team decides 128K is now the standard. A capacity plan should have a version number and a refresh date.

Third, a vocabulary upgrade. The next four modules give you that vocabulary. M3 explains what is actually happening inside a GPU during inference. M4 covers the KV cache, the piece of working memory that drives most of the surprises. M5 names the seven parameters you need to plan against. M6 walks them through a worked example.

Once you have the vocabulary, the planning conversation with the AI team stops feeling like translation and starts feeling like collaboration. That is the right end state, and it's the only one that holds up over time.
