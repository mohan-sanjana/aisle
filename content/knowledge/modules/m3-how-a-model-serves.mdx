
Your AI team has decided on a model, and you have it ready to go. But where do you start? And what really happens from the moment it is loaded into the GPU? This module walks you through the end-to-end lifecycle of an inference request, from a cold model file sitting on a storage array to the individual words streaming back to a user's screen.

## Phase 0: Ingestion and Routing

An LLM starts its operational life as a collection of massive files containing tens or hundreds of gigabytes of frozen mathematical parameters called weights. This is the output of the expensive training and fine-tuning processes that either a frontier model provider like OpenAI or your own internal AI team went through. These files typically live on a high-speed, parallel network filesystem like Weka or local NVMe storage tiers.

At startup, the inference server streams these weights across the network and loads them directly into the GPU's High Bandwidth Memory (<GlossaryTerm id="hbm">HBM</GlossaryTerm>). If a model is too large to fit into a single GPU's memory footprint, it is split across multiple cards using a hardware-level parallelism framework. Once the weights are completely loaded, the model is initialized and sits waiting in a "frozen" state.

When a user types a prompt (such as "What does our parental leave policy say about adoption?"), the request doesn't go straight to the hardware. It first hits an API gateway for authentication, rate limiting, and global routing. From there, the gateway routes the text to an inference engine cluster. Before the GPU can process it, a software component called a tokenizer translates the raw English text into a sequence of numbers called tokens. Roughly, one token equals three or four characters of English text. Once tokenized, this array of numbers is sent over the PCIe bus or high-speed network fabric to the specific GPUs hosting the model.

What happens next is the core mechanics of AI inference, split into two completely distinct phases.

## Phase 1: Prefill (Processing the Prompt)

The first thing the GPU must do with an incoming prompt is read it entirely to compute what every token means in this specific context. The word "leave" in "parental leave" carries an entirely different mathematical meaning than the word "leave" in "what time should we leave".

This phase is called <GlossaryTerm id="prefill">prefill</GlossaryTerm>. Think of an F1 pit crew changing all four tires on a car simultaneously. The crew doesn't wait to finish the front-left tire before starting the rear-right; they flood the car with mechanics to get the entire job done at once. Similarly, prefill processes the incoming prompt matrix in parallel, completely saturating the thousands of tiny compute cores inside the GPU. A 1,500-token prompt on a 70B model might clear the prefill phase in just 200 milliseconds.

But there's an infrastructure catch. While highly parallelized and fast, prefill is heavily bound by the GPU's raw compute power (FLOPS). Long prompts put massive immediate strain on your GPU cores. Crucially, prefill is also the exact phase that builds up the active working memory for the rest of the request, known as the <GlossaryTerm id="kv_cache">KV cache</GlossaryTerm>. The brief wall-clock delay during prefill is what users experience as Time to First Token (<GlossaryTerm id="ttft">TTFT</GlossaryTerm>).

## Phase 2: Decode (Generating the Answer)

Once the model understands the prompt, it switches to a completely different operational profile to generate the response. It does this one single token at a time. This phase is called <GlossaryTerm id="decode">decode</GlossaryTerm>.

Think of decode like an old-school library copy clerk who needs to write a new essay based on an entire room full of reference books. To write just the *first* word, the clerk has to sprint down the aisles, skim every single page of every book in the room, write that one word down on a notepad, and then walk back to the start. To write the *second* word, the clerk has to run down the aisles and skim every single book all over again, plus the new word they just wrote down.

This is exactly how inefficient the hardware process is. To generate just *one* single token, the GPU must sweep its entire 70GB+ model parameters and the conversation's active KV cache across its internal memory bus, pass it through the compute units, and write the result back. Then it repeats that entire 70GB sweep for the next word, and the next.

Because of this constant sweeping, the GPU's massive compute units sit mostly idle during decode; they are constantly waiting for data to arrive from memory. Therefore, decode throughput is bound entirely by **memory bandwidth**, not compute FLOPS. This is why an NVIDIA H200 delivers vastly better decode throughput than an H100; it isn't faster at math, it just has a 43% wider memory pipe to move those parameters past the processors. The time between consecutive output tokens is called Time per Output Token (<GlossaryTerm id="tpot">TPOT</GlossaryTerm>).

## Handling Mixed Workloads

In production, your cluster won't just handle one clean request at a time. It will get a chaotic mix of short chatbot prompts (which want fast decode speeds) and massive document uploads (which require heavy prefill compute).

Because these phases have completely opposite hardware demands, running them side-by-side can cause bottlenecks. Modern inference engines solve this using advanced scheduling techniques like **continuous batching** or **chunked prefills** to interleave requests smoothly without stalling active chat sessions. More on how these software engine choices impact physical cluster layouts later.

## A complete request, end to end

A request from the user's keyboard to the streamed response looks like this:

{`sequenceDiagram

    U->>API: POST /chat with prompt

    Srv->>GPU: prompt tokens
    Note over GPU: PREFILL: read all tokens in parallel, build KV cache
    GPU-->>Srv: first output token
    Srv-->>U: stream first token (TTFT)

    loop For each remaining output token
      Note over GPU: DECODE: read weights + KV cache from memory, generate next token
      GPU->>Srv: next token
      Srv->>U: stream token (TPOT)
    end`}
</KnowledgeMermaid>

The total wall-clock time the user waits is TTFT plus the number of output tokens multiplied by TPOT. A 300-token answer at 30 ms per token plus a 200 ms prefill takes about 9.2 seconds. Streamed back token by token, this feels natural. Delivered as a single response at the end, it would feel slow.

## Summary for IT Planners

Understanding this split tells you exactly which hardware levers to pull when collaborating with your AI development team:

1. **If your workload has long prompts and short answers** (like analyzing massive legal contracts or financial logs), your system will spend its life in **Prefill**. You need to provision for heavy **GPU Compute (FLOPS)**.
2. **If your workload has short prompts and long answers** (like interactive chatbots or code generation assistants), your system will spend its life in **Decode**. You need to prioritize **Memory Bandwidth (HBM3e/GB/s)**.
